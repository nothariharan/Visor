# VISOR 3.0: Realistic 2-Week Sprint Plan

**Reality Check:** Based on the audit, VISOR is at 40% IDE readiness. You have a solid graph foundation but lack the "IDE loops" that make developers stay.

**What we're building:** Not VS Code 2.0. We're building "VS Code's visual cousin" - same power, better navigation, unique collaboration.

**Time Budget:** 14 days × 8 hours = 112 hours of focused work (with LLM assistance: effectively ~180 hours)

---

## WEEK 1: Stop The Bleeding (Days 1-7)

**Goal:** Fix the 3 things that make users quit immediately.

### DAY 1-2: Persistent State & Layouts [24 hours]

**Current Problem:** "reload loses layout" - 100% of users will hit this. It's not a bug, it's a dealbreaker.

#### Implementation Plan

**Backend Changes (6 hours):**

1. **Create `.visor/` Directory Manager**
   - When VISOR starts, check if `.visor/` exists in project root
   - If not, create it with three JSON files:
     ```
     .visor/
     ├── layout.json
     ├── groups.json
     └── preferences.json
     ```
   - Add to `.gitignore` automatically

2. **Layout Save Endpoint**
   - Add `POST /api/visor/save-layout` endpoint
   - Accepts JSON with:
     ```json
     {
       "nodes": [/* React Flow nodes with positions */],
       "viewport": { "x": 0, "y": 0, "zoom": 1 },
       "mode": "skeleton",
       "expandedFolders": ["src", "components"],
       "hiddenNodes": ["node_modules"]
     }
     ```
   - Write to `.visor/layout.json`
   - Don't validate too strictly - just save what frontend sends

3. **Layout Load Endpoint**
   - Add `GET /api/visor/load-layout`
   - Read `.visor/layout.json`
   - If file doesn't exist, return `{ exists: false }`
   - If corrupted, return default layout (don't crash)

**Frontend Changes (10 hours):**

4. **Auto-Save Hook**
   - Create `useAutoSave.js` hook:
     ```javascript
     useEffect(() => {
       const timer = setTimeout(() => {
         saveLayout(nodes, viewport, mode);
       }, 2000); // 2 second debounce
       
       return () => clearTimeout(timer);
     }, [nodes, viewport, mode]);
     ```
   - Call on every `onNodesChange`, `onViewportChange`
   - Show "Saving..." indicator (bottom right)

5. **Restore on Load**
   - On app mount, fetch `/api/visor/load-layout`
   - If layout exists, use it as initial state
   - Use React Flow's `fitView` to animate to saved viewport
   - If layout is corrupted/old version, ignore and use defaults

6. **Manual Controls**
   - Add "Reset Layout" button in header
   - Add "Export Layout" (download JSON)
   - Add "Import Layout" (upload JSON)

**Testing (4 hours):**
- Open VISOR, move nodes around, close tab
- Reopen → nodes should be in exact same positions
- Test with 100+ nodes (should load in <2s)
- Test with corrupted JSON (should fall back gracefully)

**Validation (4 hours):**
- Add versioning to layout JSON:
  ```json
  {
    "version": "3.0.0",
    "savedAt": "2025-02-21T10:30:00Z",
    "data": { /* layout data */ }
  }
  ```
- Handle version mismatches

#### Dos and Don'ts

**DO:**
- ✅ Save frequently (every 2 seconds) but debounced
- ✅ Include a timestamp in the saved file
- ✅ Add a version number to detect outdated layouts
- ✅ Show "Last saved: 2 seconds ago" in status bar
- ✅ Handle missing `.visor/` directory gracefully

**DON'T:**
- ❌ Save on every tiny change (kills performance)
- ❌ Store absolute positions (use relative to viewport)
- ❌ Crash if layout.json is corrupted (just reset)
- ❌ Save node content (only save positions/metadata)
- ❌ Make it complicated - simple JSON is fine

**Common Pitfalls:**
- React Flow node positions are in global coordinates - don't forget to save viewport zoom/pan
- Don't serialize functions or React components
- Watch out for circular references in node data

### DAY 3-4: Universal Search (Cmd+P) [16 hours]

**Current Problem:** Search exists but feels slow. No keyboard shortcuts. Not modal.

#### Implementation Plan

**Backend Changes (4 hours):**

1. **File Index Endpoint**
   - Add `GET /api/search/files?q=query`
   - Return array of files matching query
   - Use simple fuzzy matching on backend:
     - Split query into characters
     - Check if all characters appear in order in filename
     - Score by match quality
   - Limit to top 50 results
   - Cache file list (only rebuild on file change)

2. **Recently Opened Tracking**
   - Add `POST /api/search/recent` endpoint
   - Store in `.visor/preferences.json`:
     ```json
     {
       "recentFiles": [
         { "path": "src/App.jsx", "lastOpened": 1708531200 }
       ]
     }
     ```
   - Sort by timestamp, limit to 20 most recent

**Frontend Changes (8 hours):**

3. **Search Modal Component**
   - Create `SearchModal.jsx`:
     - Full-screen overlay (dark background)
     - Centered input box
     - Results list below
     - ESC to close
   - Style like terminal (monospace, Catppuccin colors)

4. **Keyboard Shortcuts**
   - Add global listener for `Cmd+P` (Mac) or `Ctrl+P` (Windows)
   - Prevent default browser behavior
   - Focus input when modal opens
   - Arrow keys to navigate results
   - Enter to select
   - ESC to close

5. **Fuzzy Search UI**
   - As user types, query `/api/search/files?q=...`
   - Debounce by 150ms
   - Highlight matched characters:
     - Input: "apjx"
     - Result: "**Ap**p.**jsx**" (bold matched chars)
   - Show file path in gray below filename
   - Show file icon (use file extension)

6. **Recent Files**
   - When input is empty, show recent files
   - "Recent Files" header
   - Same styling as search results
   - Update list when file is opened

7. **Jump to Node**
   - When user selects file:
     - Close modal
     - Use React Flow `setCenter` to pan to node
     - Highlight node with pulse animation
     - Open file in Monaco editor

**Testing (2 hours):**
- Type 3 characters → should find file in <100ms
- Test with 500+ files
- Test keyboard navigation (arrows, enter, esc)
- Test on Mac and Windows (different key combos)

**Polish (2 hours):**
- Add loading spinner while searching
- Show "No results" message
- Show file count in results
- Add icons for file types

#### Dos and Don'ts

**DO:**
- ✅ Make search instant (<100ms)
- ✅ Show recent files when empty
- ✅ Highlight matched characters
- ✅ Support fuzzy matching ("apjx" → "App.jsx")
- ✅ Use global keyboard shortcut

**DON'T:**
- ❌ Search file contents (too slow, do that later)
- ❌ Use complex fuzzy libraries (simple algorithm is fine)
- ❌ Search node_modules (exclude automatically)
- ❌ Make users click to focus input (auto-focus)
- ❌ Search on every keystroke (debounce 150ms)

**Common Pitfalls:**
- Mac uses `metaKey`, Windows uses `ctrlKey`
- Prevent default on Cmd+P or browser opens print dialog
- React Flow center animation is async - wait for it
- Remember to stop event propagation

### DAY 6-7: Terminal Integration [16 hours]

**Current Problem:** Terminal is read-only. No stdin. Can't type commands.

#### Implementation Plan

**Backend Changes (8 hours):**

1. **Install node-pty**
   ```bash
   npm install node-pty
   ```

2. **Replace spawn with pty**
   - Change `ProcessRunner.js`:
     ```javascript
     import pty from 'node-pty';
     
     const ptyProcess = pty.spawn(shell, [], {
       name: 'xterm-color',
       cwd: workingDir,
       env: process.env
     });
     ```
   - This gives you a real TTY (can handle stdin)

3. **WebSocket for Terminal I/O**
   - Add Socket.io event for terminal:
     ```javascript
     socket.on('terminal:input', (data) => {
       ptyProcess.write(data);
     });
     
     ptyProcess.onData((data) => {
       socket.emit('terminal:output', data);
     });
     ```

**Frontend Changes (6 hours):**

4. **Install xterm.js**
   ```bash
   npm install xterm xterm-addon-fit
   ```

5. **Create Terminal Component**
   - Replace current read-only terminal with xterm.js:
     ```javascript
     import { Terminal } from 'xterm';
     
     const term = new Terminal();
     term.open(containerRef.current);
     
     term.onData((data) => {
       socket.emit('terminal:input', data);
     });
     
     socket.on('terminal:output', (data) => {
       term.write(data);
     });
     ```

6. **Terminal Tabs**
   - Allow multiple terminals
   - + button to create new terminal
   - Each terminal is a separate pty process

**Testing (2 hours):**
- Type `ls` → should see output
- Type `vim` → should work (full TTY)
- Test colors (ANSI codes)
- Test Ctrl+C (interrupt)

#### Dos and Don'ts

**DO:**
- ✅ Use node-pty (it's the standard)
- ✅ Use xterm.js (it's battle-tested)
- ✅ Support multiple terminal tabs
- ✅ Auto-resize terminal on panel resize

**DON'T:**
- ❌ Build your own terminal emulator
- ❌ Try to parse ANSI codes yourself
- ❌ Make terminal full-screen yet (keep bottom panel)
- ❌ Add tmux/screen integration (overkill)

**Common Pitfalls:**
- node-pty requires native compilation (needs build tools)
- Windows uses different shell (cmd.exe or powershell)
- xterm.js needs explicit resize calls
- Don't forget to kill pty process on close

### DAY 10-11: LSP Integration (Basic) [16 hours]

**Current Problem:** No IntelliSense, no error squiggles, feels dumb.

**Reality Check:** Full LSP is 40+ hours. We're doing MVP only.

#### Implementation Plan

**Backend LSP Proxy (10 hours):**

1. **Install Dependencies**
   ```bash
   npm install vscode-ws-jsonrpc vscode-languageserver-protocol
   ```

2. **Spawn TypeScript Language Server**
   - Create `lsp/server.js`:
     ```javascript
     import { spawn } from 'child_process';
     
     const tsserver = spawn('typescript-language-server', [
       '--stdio'
     ]);
     ```

3. **WebSocket Bridge**
   - Proxy LSP messages between Monaco and language server:
     ```javascript
     socket.on('lsp:request', (message) => {
       tsserver.stdin.write(JSON.stringify(message));
     });
     
     tsserver.stdout.on('data', (data) => {
       socket.emit('lsp:response', JSON.parse(data));
     });
     ```

**Frontend Integration (4 hours):**

4. **Connect Monaco to LSP**
   - Use `monaco-languageclient`:
     ```javascript
     import { MonacoLanguageClient } from 'monaco-languageclient';
     
     const client = new MonacoLanguageClient({
       name: 'TypeScript',
       clientOptions: { /* ... */ }
     });
     ```

5. **Enable LSP Features**
   - Monaco automatically gets:
     - Red squiggles on errors
     - Autocomplete dropdown
     - Hover tooltips
     - Go to definition

**Testing (2 hours):**
- Open TypeScript file
- Type wrong code → see red squiggle
- Trigger autocomplete → see suggestions
- Ctrl+Click variable → jump to definition

#### Dos and Don'ts

**DO:**
- ✅ Start with TypeScript only (one language)
- ✅ Use existing LSP servers (typescript-language-server)
- ✅ Use monaco-languageclient (don't build from scratch)
- ✅ Test with real projects

**DON'T:**
- ❌ Try to support 10 languages in 2 days
- ❌ Build your own language server
- ❌ Try to implement all LSP features
- ❌ Worry about performance yet

**Common Pitfalls:**
- LSP servers expect file:// URIs not absolute paths
- Monaco needs workspace folders configured
- Some LSP servers crash on invalid JSON
- Windows paths need special handling

**MVP Scope:**
- ✅ Error detection (red squiggles)
- ✅ Basic autocomplete
- ✅ Hover for type info
- ❌ Rename refactoring (too complex)
- ❌ Code actions (later)
- ❌ Multiple languages (TypeScript only)
